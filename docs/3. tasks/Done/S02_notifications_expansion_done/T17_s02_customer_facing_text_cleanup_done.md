# T17 — Customer-facing правки текстов, Wazzup-шаблонов и send window S02

## Customer-facing инкремент

Клиент получает сообщения S02 в WhatsApp с согласованной бизнес-формулировкой, без артефактов рендеринга и опечаток, а сервис отправляет их в окне 08:00-22:00 Europe/Berlin.

## Контекст

- T16 закрыла проблему доставляемости: Б1/Б2/Б4 переведены на UTILITY и серия Б1-Б5 стала рабочим baseline.
- После просмотра реальных чатов выявлены customer-facing дефекты в текстах и Wazzup-шаблонах:
  - в Г1 `gosniki_consultation_done` используется формулировка `заказ`, но в клиентском тексте должно быть `запрос`;
  - Б1 сейчас приходит как операционное уведомление о записи, а целевой тон — поздравительный onboarding-текст;
  - в сообщениях встречаются `Jobcenter`, `Agentur fur Arbeit`, `AA`, хотя клиентский текст должен быть унифицирован;
  - в шаблонах/подстановках видны `назначенное время`, лишнее `в` перед датой, опечатки `в этот` / `Хотеди`;
  - списки с checkbox/bullet-символами рендерятся квадратами/точками в Wazzup/WhatsApp.
- На боевом тесте сделки `18143828` дополнительно обнаружен дефект orchestration:
  - после сдвига даты АА на `+3 дня` клиент получил два разных customer-facing сообщения почти подряд;
  - `berater_accepted` ушёл повторно через retry/pending-механику, а следом корректно сработал temporal `berater_day_minus_3`;
  - итоговая задача должна устранить такие наложения, чтобы клиент не получал серию из старого webhook-сообщения и нового temporal-сообщения на одно и то же актуальное состояние сделки.
- Доступ к Wazzup получен, поэтому задача включает не только кодовую подстановку, но и фактическую правку/замену шаблонов.

## Критерии приёмки

1. Для Г1 customer-facing формулировка использует `запрос`, а не `заказ`.
2. Для согласованных линий S02 клиент больше не видит `Jobcenter`, `Agentur für Arbeit` или `AA`; используется единая customer-facing формулировка `с Бератором`.
3. Б1 больше не отправляется в текущей формулировке `Вы записаны...`; вместо неё используется согласованный поздравительный текст.
4. В customer-facing тексте S02 отсутствует строка `назначенное время`.
5. В сообщениях с датой устранён лишний предлог `в` перед датой там, где он дублируется шаблоном.
6. Чек-листы и списки не рендерятся квадратами или неконтролируемыми bullet-символами в Wazzup/WhatsApp.
7. Исправлены зафиксированные текстовые дефекты `в этот` → `в это` и `Хотеди` → `Хотели`.
8. Окно отправки сервиса изменено на `08:00-22:00 Europe/Berlin`, и тесты/документация это отражают.
9. Wazzup template mapping, retry/backfill-совместимость и документация синхронизированы с новыми текстами.
10. Для `berater_day_minus_7` по АА реализовано правило `дата + этап`: сообщение не уходит только по одной дате, если сделка не стоит в `102183943` или `102183947`.
11. При переносе даты сделки сервис не досылает устаревший `berater_accepted` рядом с новым temporal-сообщением (`berater_day_minus_3` / `berater_day_minus_1` / `berater_day_0`), если для клиента уже актуально более позднее состояние.

## Риски и ограничения

- Поздравительный Б1-текст может потребовать новый шаблон или другую категорию Wazzup. Это нельзя внедрять в прод без проверки фактической доставляемости, иначе можно отменить выигрыш T16.
- Часть формулировок может быть зашита прямо в теле Wazzup-шаблона, поэтому кодовой заменой `templateValues` задача не ограничивается.
- Изменение send window глобальное для сервиса: оно затрагивает и S01-линии, хотя задача оформлена в рамках S02 по пользовательскому запросу.
