## Тест-план

### Сценарий 1: Первая линия (запись на термин)

**Подготовка:**
1. Создать тестовый контакт в Kommo:
   - Имя: "Тест Иван Первая"
   - Телефон: +996501354144
   - Дата Термина: завтра 14:00
   - Воронка: "Бератер"
   - Этап: любой начальный

**Шаги:**
1. Переместить контакт на этап "Принято от первой линии" (first)
2. Дождаться отправки webhook от Kommo (1-5 сек)

**Ожидаемый результат:**
- ✅ WhatsApp-сообщение приходит на +996501354144
- ✅ Текст содержит: имя ("Иван"), дату ("завтра в 14:00"), призыв записаться
- ✅ В Kommo добавлено примечание: "WhatsApp сообщение отправлено: YYYY-MM-DD HH:MM:SS"
- ✅ В БД создана запись: `status=sent`, `line=first`, `attempts=1`

### Сценарий 3: Отправка вне окна времени (pending)

**Подготовка:**
1. Установить на сервере `SEND_WINDOW_END=14` (окно до 14:00)
2. Подождать до 15:00

**Шаги:**
1. Переместить тестовый контакт на триггерный этап

**Ожидаемый результат:**
- ✅ Сообщение НЕ отправлено сразу
- ✅ В БД: `status=pending`, `next_retry_at` = завтра 9:00
- ✅ В логах: "Scheduled for next send window"

**Продолжение (следующий день в 9:00):**
1. Дождаться запуска cron (или запустить вручную: `systemctl start whatsapp-cron`)

**Ожидаемый результат:**
- ✅ Сообщение отправлено
- ✅ БД: `status=sent`, `sent_at` обновлён

---

### Сценарий 4: Повторная отправка через 24ч

**Подготовка:**
1. Создать тестовую запись в БД:
   ```python
   from server.db import db
   from datetime import datetime, timedelta

   db.create_message(
       kommo_contact_id=12345,
       phone="+996501354144",
       line="first",
       message_text="Тест повтор",
       status="sent",
       messenger_backend="wazzup",
       sent_at=datetime.now() - timedelta(hours=25),
       next_retry_at=datetime.now() - timedelta(minutes=5)
   )
   ```

**Шаги:**
1. Запустить cron: `systemctl start whatsapp-cron`

**Ожидаемый результат:**
- ✅ Сообщение отправлено повторно
- ✅ БД: `attempts=2`, `sent_at` обновлён, `next_retry_at` = now + 24ч
- ✅ Логи cron: "Retry successful"

**Продолжение (через 24ч):**
1. Запустить cron ещё раз

**Ожидаемый результат:**
- ✅ Третья попытка (attempts=3)
- ✅ Больше повторов не будет (макс достигнут)

### Сценарий 6: Ошибка Kommo API (несуществующий lead)

**Подготовка:**
1. Отправить webhook с невалидным `lead_id=99999999`

```bash
curl -X POST https://whatsapp.sternmeister.com/webhook/kommo \
  -H "Content-Type: application/json" \
  -d '{
    "leads": {
      "status": [{
        "id": 99999999,
        "status_id": 67890,
        "pipeline_id": 111,
        "old_status_id": 67889
      }]
    }
  }'
```

**Ожидаемый результат:**
- ✅ HTTP 500
- ✅ Telegram alert: "Ошибка Kommo API, Lead ID: 99999999, Ошибка: 404 Not Found"

### Сценарий 7: Переключение messenger backend

**Шаги:**
1. Установить `MESSENGER_BACKEND=green_api` в `.env`
2. Перезапустить контейнер: `docker restart whatsapp-notifications`
3. Переместить контакт на триггерный этап

**Ожидаемый результат:**
- ✅ Сообщение отправлено через Green API (произвольный текст)
- ✅ БД: `messenger_backend=green_api`

**Продолжение:**
1. Установить `MESSENGER_BACKEND=wazzup`
2. Перезапустить
3. Переместить другой контакт

**Ожидаемый результат:**
- ✅ Сообщение отправлено через Wazzup24 (WABA-шаблон с кнопками)
- ✅ БД: `messenger_backend=wazzup`

### Сценарий 8: Проверка всех DoD из S01

Пройти по всем критериям приёмки из спецификации S01 и отметить:

#### Acceptance Criteria из S01:

- [ ] Webhook от Kommo принимается корректно (валидация payload)
- [ ] Сообщение отправляется в WhatsApp при смене этапа воронки "Бератер" (первая/вторая линия)
- [ ] Персонализация работает: имя клиента + дата термина
- [ ] Отправка происходит только в окне 9:00–21:00 (CET/CEST)
- [ ] Сообщения вне окна откладываются до 9:00 следующего дня
- [ ] Повторная отправка работает: через 24ч при отсутствии ответа, максимум 2 повтора
- [ ] Примечание "WhatsApp сообщение отправлено" записывается в Kommo
- [ ] Telegram-алерт отправляется при ошибке (Green API недоступен, невалидный номер и т.д.)
- [ ] Все события логируются в SQLite (status, attempts, timestamps)
- [ ] Messenger layer поддерживает переключение Green API ↔ Wazzup24 через `MESSENGER_BACKEND` в .env
- [ ] Wazzup24 использует WABA-шаблон "Напоминание о записи или встрече"
- [ ] Cron-задача запускается каждый час и обрабатывает повторы + отложенные сообщения
- [ ] Проект запускается в Docker на сервере Hetzner (65.108.154.202)
- [ ] `.env.example` содержит все необходимые переменные окружения
