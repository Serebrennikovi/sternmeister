# Гайд: как писать спецификации

**Версия:** 1.3
**Дата создания:** 26 декабря 2025
**Дата обновления:** 11 января 2026
**Статус:** Действует

---

## 1. Назначение спецификации

Спецификация — это **разработческий документ**, который отвечает на вопросы:

* **Что именно нужно построить** — какой функционал, экраны, поведение
* **Зачем это важно сейчас** — какой инкремент получает пользователь или бизнес
* **Какие границы реализации** — что входит и что явно не входит в scope
* **Как это устроено** — архитектура, структура, интеграции
* **Как проверяем результат** — критерии приёмки (DoD)

---

## 2. Типы спецификаций

### 2.1. UX/UI-спецификации (первый этап)

**Назначение:** зафиксировать структуру интерфейса, поведение пользователя, подготовить основу для вёрстки.

**Содержит:**
* Структуру экранов и навигацию
* User Flow и сценарии использования
* Поведенческую логику и UX-принципы
* Список данных, необходимых на фронтенде
* Референсы подхода (без копирования визуала)

**НЕ содержит:**
* Детальный визуальный дизайн (это Figma)
* Backend-архитектуру (это отдельная спека)
* Готовые UI-макеты из других проектов

### 2.2. Функциональные спецификации (второй этап)

**Назначение:** описать интеграцию фронтенда с backend и внешними сервисами.

**Содержит:**
* API endpoints и контракты
* Авторизацию и безопасность
* Модели данных и БД
* Логику обработки на backend
* Интеграции с внешними сервисами
* Edge Functions / Serverless функции

## 3. Стандартная структура UX/UI-спецификации

### Шаблон имени файла

> См. [doc_conventions.md](doc_conventions.md) — единый формат для всех спецификаций

```
SNN_<snake_case_название>.md
```
Примеры: `TGB-S01_feed_redesign.md`, `TGB-S02_message_composer.md`

---

### 3.1. Заголовок

```markdown
# UX/UI Спецификация: <Название фичи>

**ID:** SNN
**Статус:** draft / active / done / deprecated
**Версия:** X.X
**Дата:** YYYY-MM-DD
**Автор:** <Имя>
```

---

### 3.2. Цель фичи

1–2 абзаца:
* Что добавляем/меняем в приложении
* Какую задачу пользователя решаем
* Почему это важно на текущем этапе

**Пример:**
```
Редизайн ленты постов позволит пользователям быстрее находить релевантный
контент благодаря улучшенной фильтрации и бесконечному скроллу. Это первый
экран после авторизации, который задаёт общее впечатление от платформы.
```

---

### 3.3. Scope

#### Входит:
* Конкретный перечень экранов, компонентов, функциональности

#### Не входит / Out of scope:
* Всё, что точно не делаем на этом этапе
* Будущие идеи, которые могут отвлечь разработчика

**Пример:**
```markdown
### Входит:
- Экран ленты с infinite scroll
- Фильтр по типу контента (посты/статьи/видео)
- Skeleton loader при загрузке
- Pull-to-refresh на мобильных

### Не входит:
- Редактирование постов (будет в S03)
- Аналитика просмотров (будет в S05)
- Push-уведомления о новых постах
```

---

### 3.4. User Flow

Краткое описание пути пользователя:
* Откуда приходит
* Что делает
* Куда переходит
* Какие состояния видит

**Формат:** последовательность шагов или диаграмма (текстом).

**Пример:**
```
1. Пользователь авторизуется → попадает на экран ленты
2. Видит карточки постов с базовой информацией
3. Может применить фильтр по типу контента (вкладки)
4. Скролл вниз → автоматическая подгрузка следующей страницы
5. Тап на карточку → переход к детальному просмотру поста
6. Pull-to-refresh → обновление ленты
```

---

### 3.5. Структура экранов

Для каждого экрана описать:

#### Экран: <Название>

**Назначение:**
Зачем нужен этот экран.

**Композиция блоков:**
* Header (шапка)
* Content (основной контент)
* Footer / Navigation (навигация)

**Интерактивные элементы:**
* Кнопки, ссылки, инпуты
* Их поведение при взаимодействии

**Состояния:**
* **Loading** — что видит пользователь при загрузке
* **Empty** — что показываем, если данных нет
* **Error** — как отображаем ошибки
* **Success** — нормальное состояние с данными

**Переходы:**
Куда можно перейти с этого экрана.

---

### 3.6. UX-принципы и поведенческая логика

Описать ключевые принципы UX для этой фичи:
* Минимальное количество действий
* Быстрый доступ к нужным данным
* Ожидаемое поведение (скролл, фильтры, свайпы)

**Пример:**
```
- **Мгновенная фильтрация:** переключение вкладок не вызывает перезагрузку страницы
- **Skeleton loader:** показываем заглушки вместо спиннера для лучшего UX
- **Optimistic updates:** лайк применяется мгновенно, откат при ошибке
- **Infinite scroll:** подгрузка начинается за 3 экрана до конца списка
```

---

### 3.7. Данные на фронтенде

Описать, какие данные нужны для отображения экранов.
**НЕ моделировать API**, а просто перечислить сущности и их атрибуты.

**Пример:**
```markdown
### Сущность: Post (карточка поста)

Необходимые поля:
- id: уникальный идентификатор
- author: объект автора (id, name, avatar)
- content: текст поста
- images: массив URL изображений
- created_at: дата создания
- likes_count: количество лайков
- comments_count: количество комментариев
- is_liked: лайкнул ли текущий пользователь

### Сущность: Filter
- active_filter: "all" | "posts" | "articles" | "videos"
```

---

### 3.8. Референсы подхода

**Важно:** не копируем визуал, а описываем **жанр интерфейсных решений**.

**Пример:**
```
- Подход к отображению: карточная сетка с краткой информацией
- Фильтрация: вкладки сверху (tabs), а не dropdown
- Навигация: bottom navigation bar
- Интерактивность: минимум анимаций, фокус на контенте
```

---

### 3.9. Ограничения и допущения

Явно зафиксировать:
* Технические ограничения
* UX-допущения
* Зависимости от других компонентов

**Пример:**
```
- Предполагаем, что лента содержит до 1000 постов (пагинация обязательна)
- Mobile-first: адаптивный дизайн с приоритетом мобильных устройств
- Требуется авторизация для просмотра ленты
```

---

### 3.10. Требования к backend (для следующего этапа)

Краткий список того, что потребуется от backend.
Это **НЕ детальная архитектура**, а просто список требований.

**Пример:**
```
- API endpoint для получения списка постов с пагинацией
- Фильтрация постов по типу на backend
- Авторизация через JWT токен
- Real-time обновления при новых постах (опционально)
```

---

### 3.11. Acceptance Criteria / DoD

Список критериев, по которым можно считать фичу готовой.

**Пример:**
```
- [ ] Экран ленты отображается корректно на iOS/Android/Desktop
- [ ] Skeleton loader показывается при загрузке
- [ ] Фильтрация работает без перезагрузки страницы
- [ ] Infinite scroll подгружает следующую страницу
- [ ] Empty state отображается, если постов нет
- [ ] Error state показывается при ошибке API
- [ ] Pull-to-refresh работает на мобильных
- [ ] Интерфейс адаптивен (работает на разных размерах экрана)
```

---

### 3.12. Связанные документы

Ссылки на:
* Бизнес-требования (BR)
* Другие спецификации (зависимости)
* Задачи в трекере

**Пример:**
```
- BR: `Docs/1. business requirements/BR_01_feed_improvements.md`
- Зависит от: `S03_auth.md` (авторизация)
- Блокирует: `S02_post_details.md` (детальная страница)
```

**Версия:** X.X

### 4.2. Цель

1–2 абзаца:
* Какой функционал реализуем
* Какую проблему решаем
* Как это вписывается в общую архитектуру

---

### 4.3. Scope

Аналогично UX/UI-спецификации: что входит, что не входит.

### 4.4. Архитектура и структура

Описать:
* Какие модули/компоненты создаются или изменяются
* Как это встраивается в существующую систему
* Диаграммы взаимодействия (если нужно)

---

### 4.5. Модели данных / БД

Описать:
* Какие таблицы/модели создаются/изменяются
* Какие поля добавляются
* Какие индексы нужны
* Связи между таблицами

---

### 4.6. API Endpoints

Для каждого endpoint описать:
* Метод и путь
* Параметры (query, path, body)
* Формат ответа
* Коды ошибок
* Правила авторизации

**Пример:**
```markdown
### GET /api/posts

**Назначение:** Получение списка постов с пагинацией

**Query параметры:**
- page: number (default: 1)
- limit: number (default: 20, max: 100)
- type: "all" | "posts" | "articles" | "videos"

**Response 200:**
```json
{
  "data": [...],
  "pagination": {
    "page": 1,
    "limit": 20,
    "total": 150,
    "has_more": true
  }
}
```

**Response 401:** Не авторизован
**Response 500:** Внутренняя ошибка сервера
```

---

### 4.7. Авторизация и безопасность

Обязательная секция.

Описать:
* Как происходит авторизация
* Где хранится токен на фронте
* Rate limiting (если нужен)
* Защита от XSS, CSRF

---

### 4.8. Логика и алгоритмы

Если есть сложная бизнес-логика, описать её.

---

### 4.9. Acceptance Criteria / DoD

Список критериев готовности функционала.

---

### 4.10. Тест-план

Если задача сложная, описать сценарии тестирования.

---

### 4.11. Зависимости и интеграции

Указать:
* Что должно быть готово до начала разработки
* Какие другие спецификации зависят от этой
* Порядок реализации

---

### 4.12. Риски и ограничения

Зафиксировать потенциальные проблемы.

---

### 4.13. Связанные документы

Аналогично UX/UI-спецификации.

## 5. Принципы хорошей спецификации

### 5.1. Строгое разделение "что строим" и "как проверяем"
Scope ≠ Acceptance Criteria.
Scope описывает функционал, DoD — критерии готовности.

### 5.2. Reuse-first подход
Всегда писать:
* Что переиспользуем из существующей системы
* Что рефакторим
* Что удаляем/заменяем

### 5.3. Никакого UI-хардкода (для UX/UI-спек)
Только **поведение и структура**, а не пиксели и цвета.

### 5.4. Никакого избыточного "почему" — это в BR
Спецификация должна быть короткой и инженерной.
Подробное обоснование — в бизнес-требованиях.

### 5.5. Одна спека = один инкремент
Идеально, если после реализации одной спеки можно:
* Залить в прод
* Показать пользователю
* Получить фидбек

### 5.6. Стиль: точный, сухой, инженерный
Без лирики, без "наверное", без "возможно".
Конкретные решения и чёткие критерии.

### 5.7. Fail fast: никаких дефолтных значений
Если данных нет — падаем с ошибкой, а не подставляем значения по умолчанию.

### 5.8. Observability: логируем всё важное
Каждая ошибка, каждое важное событие — в лог с контекстом.

## 9. Рекомендации по работе со спецификациями

### 9.1. Версионирование

При изменении спецификации:
1. Обновите **Версию** (например, 1.0 → 1.1)
2. Укажите **Дату** изменения
3. Добавьте секцию **История изменений** в конце документа:

```markdown
## История изменений

### v1.1 (2025-12-27)
- Добавлен экран "Статистика"
- Уточнены требования к API endpoint

### v1.0 (2025-12-26)
- Первая версия спецификации
```

### 9.2. Статусы спецификации

> См. [doc_conventions.md](doc_conventions.md) — полное описание статусов

- **draft** — черновик, в разработке
- **active** — действующая, по ней ведется работа
- **done** — завершенная, все задачи выполнены
- **deprecated** — устаревшая, заменена другой спецификацией

### 9.3. Naming conventions

> См. [doc_conventions.md](doc_conventions.md) — единый источник правды

**Все спецификации (единый формат):**
```
TGB-S01_feed_redesign.md
TGB-S02_message_composer.md
S03_auth.md
S04_posts_api.md
```

### 9.4. Где хранить

```
Docs/

│   ├── TGB-S01_feed_redesign.md
│   ├── TGB-S02_message_composer.md
│   ├── S03_auth.md

## 10. Заключение

Этот гайд поможет команде создавать **единообразные, чёткие и полные спецификации**.

**Ключевые принципы:**
- Одна спека = один инкремент
