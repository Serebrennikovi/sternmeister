## T00 — Сбор конфигурации из Kommo CRM

**Дата:** 2026-02-23
**Статус:** draft
**Спецификация:** S01_whatsapp_auto_notifications.md

**⚠️ ОТСТУПЛЕНИЕ ОТ ГАЙДА СОГЛАСОВАНО:**
Это техническая задача без customer-facing инкремента. Выделена отдельно как prerequisites для всех функциональных задач T01-T09. Без реальных ID из Kommo CRM невозможно реализовать ни одну функциональную задачу.

### Customer-facing инкремент

Разработчик имеет полный набор реальных идентификаторов (pipeline_id, status_id, field_id) из Kommo CRM для реализации системы уведомлений.

### Scope

**Входит:**
- Получить `pipeline_id` для воронок "Бератер" и "Госники"
- Получить `status_id` для триггерных этапов:
  - "Принято от первой линии" (первая линия)
  - "Термин ДЦ назначен" (вторая линия)
- Получить `field_id` для кастомного поля "Дата Термина"
- Создать тестовый контакт в Kommo:
  - Имя: "Тест Иван"
  - Телефон: +996501354144
  - Воронка: "Бератер"
- Документировать реальную структуру API ответов (JSON примеры)
- Создать `server/config.py` с константами

**Не входит:**
- Написание кода webhook handler
- Интеграция с Wazzup24 (будет в T02)
- Настройка webhook URL в Kommo

### Как протестировать

1. **Получить pipeline_id:**
   ```bash
   curl -H "Authorization: Bearer $KOMMO_TOKEN" \
     https://sternmeister.kommo.com/api/v4/leads/pipelines
   ```
   → Найти воронки "Бератер" и "Госники", записать их ID

2. **Получить status_id:**
   - В ответе выше найти этапы "Принято от первой линии" и "Термин ДЦ назначен"
   - Записать их ID

3. **Получить field_id:**
   ```bash
   curl -H "Authorization: Bearer $KOMMO_TOKEN" \
     https://sternmeister.kommo.com/api/v4/leads/custom_fields
   ```
   → Найти поле "Дата Термина", записать field_id

4. **Создать тестовый контакт:**
   - Открыть Kommo UI
   - Создать контакт "Тест Иван" с телефоном +996501354144
   - Добавить в воронку "Бератер"
   - Заполнить "Дата Термина": 2026-02-25 14:00
   - Записать lead_id

5. **Документировать API ответ:**
   ```bash
   curl -H "Authorization: Bearer $KOMMO_TOKEN" \
     "https://sternmeister.kommo.com/api/v4/leads/{lead_id}?with=contacts"
   ```
   → Сохранить полный JSON в `docs/examples/kommo_lead_response.json`

### Критерии приёмки

- [ ] Файл `server/config.py` создан со всеми константами:
  ```python
  PIPELINE_CONFIG = {
      <real_id>: {  # Воронка "Бератер"
          <real_status_id>: "first",   # "Принято от первой линии"
          <real_status_id>: "second",  # "Термин ДЦ назначен"
      },
      <real_id>: {  # Воронка "Госники"
          # ...
      }
  }

  TERMIN_DATE_FIELD_ID = <real_field_id>
  ```

- [ ] Тестовый контакт создан в Kommo (lead_id записан в config.py или README)
- [ ] Файл `docs/examples/kommo_lead_response.json` содержит реальный пример ответа
- [ ] Документ обновлён: в S01 заменены placeholder значения (111, 67890, etc) на реальные

### Зависимости

- **Требует:** Доступ к Kommo CRM (OAuth токен)
- **Блокирует:** T01-T09

## T01 — Scaffold проекта и SQLite

**Дата:** 2026-02-23
**Статус:** draft
**Спецификация:** S01_whatsapp_auto_notifications.md

### Customer-facing инкремент

Базовая структура проекта готова: можно запустить сервис локально, база данных создаётся автоматически, есть health-check endpoint для проверки работоспособности.

### Scope

**Входит:**
- Scaffold проекта:
  - Структура папок (`server/`, `messenger/`, `docs/examples/`)
  - `requirements.txt` (Flask/FastAPI, requests, sqlite3, pytz)
  - `.env.example` с комментариями для всех переменных окружения
  - `server/app.py` с базовым Flask/FastAPI приложением
  - Health-check endpoint `GET /health` → `{"status": "ok"}`
- SQLite база данных:
  - Таблица `messages` (все поля из спеки: id, kommo_contact_id, phone, line, message_text, status, attempts, created_at, sent_at, next_retry_at, messenger_id, messenger_backend)
  - Индексы `idx_status_next_retry`, `idx_kommo_contact`
  - Файл `server/db.py` с функциями:
    - `init_db()` — создание таблицы при старте
    - `create_message(...)` — вставка записи
    - `get_messages(filters)` — выборка с фильтрами
    - `update_message(id, fields)` — обновление полей
- Использование `config.py` из T00:
  - Импорт PIPELINE_CONFIG для определения линий
  - Импорт TERMIN_DATE_FIELD_ID для извлечения даты
- Базовая функция `determine_line(status_id)` → "first"/"second"/None

**Не входит:**
- Webhook endpoint POST /webhook/kommo (будет в T02)
- Интеграция с Wazzup24 (будет в T02)
- Интеграция с Kommo API (будет в T03)
- Cron-задача (будет создана в T04)
- Персонализация, Telegram алерты

### Как протестировать

1. **Создать виртуальное окружение и установить зависимости:**
   ```bash
   cd server
   python -m venv venv
   source venv/bin/activate  # или venv\Scripts\activate на Windows
   pip install -r requirements.txt
   ```

2. **Настроить переменные окружения:**
   ```bash
   cp .env.example .env
   # Можно оставить .env пустым для этой задачи
   ```

3. **Запустить сервис:**
   ```bash
   python app.py
   ```
   → Сервис запускается на http://localhost:5000
   → В логах: "Database initialized" или "Table 'messages' already exists"

4. **Проверить health-check:**
   ```bash
   curl http://localhost:5000/health
   ```
   → Ответ: `{"status": "ok"}`

5. **Проверить SQLite:**
   ```bash
   sqlite3 messages.db
   .schema messages
   .quit
   ```
   → Таблица создана с правильными полями и индексами

6. **Проверить determine_line():**
   ```python
   from server.app import determine_line
   from server.config import PIPELINE_CONFIG

   # Тест с реальными ID из config.py
   print(determine_line(12345))  # → "first" или "second" или None
   ```

### Критерии приёмки

- [ ] Проект запускается локально без ошибок
- [ ] SQLite таблица `messages` создаётся автоматически при старте
- [ ] Индексы `idx_status_next_retry` и `idx_kommo_contact` созданы
- [ ] Health-check endpoint `GET /health` возвращает 200 OK
- [ ] Функции `create_message()`, `get_messages()`, `update_message()` работают
- [ ] `determine_line()` корректно использует PIPELINE_CONFIG из T00
- [ ] `.env.example` содержит все переменные с комментариями
- [ ] `requirements.txt` содержит все необходимые зависимости

### Зависимости

- **Требует:** T00 (config.py с PIPELINE_CONFIG)
- **Блокирует:** T02-T09

## T02 — Первое уведомление в WhatsApp через Wazzup24 WABA

**Дата:** 2026-02-23
**Статус:** draft
**Спецификация:** S01_whatsapp_auto_notifications.md

### Customer-facing инкремент

Менеджер получает уведомление в WhatsApp при смене тестового контакта на этап "Принято от первой линии" (воронка "Бератер"). Сообщение отправляется с официального номера Sternmeister (+49 3046690188) через WABA-шаблон (пока без персонализации).

### Scope

**Входит:**
- **Messenger layer архитектура** (с расчётом на будущие каналы):
  - `server/messenger/base.py` — `BaseMessenger` (абстрактный класс)
  - `server/messenger/wazzup.py` — `WazzupMessenger(BaseMessenger)`
  - Метод `send_message(phone: str, data: dict)` где `data = {"name": str, "termin_date": str, "line": str}`

- **Wazzup24 WABA интеграция:**
  - POST `https://api.wazzup24.com/v3/message`
  - Использование шаблона "Напоминание о записи или встрече" (guid: `38194e93-e926-4826-babe-19032e0bd74c`)
  - Подстановка параметров: `[[SternMeister]]; [[термине]]; [[скоро]]` (имя и дата будут в T03)
  - Переменные окружения: `WAZZUP_API_KEY`, `WAZZUP_CHANNEL_ID`, `WAZZUP_TEMPLATE_GUID`

- **Webhook endpoint** `POST /webhook/kommo`:
  - Валидация payload (проверка структуры: `leads.status[0]`)
  - Извлечение `lead_id`, `status_id`, `pipeline_id`
  - Использование `determine_line(status_id)` из T01
  - Если линия не определена → 200 OK, "Status not relevant"
  - Формирование `data` для отправки (пока хардкод: `name="Клиент"`, `termin_date="скоро"`)
  - Хардкод телефона для тестов: `+996501354144` (из T00)
  - Запись в SQLite со статусом "sent"

- **Логирование:**
  - Создание записи в SQLite через `create_message()` из T01
  - Поля: `kommo_contact_id=lead_id`, `phone=+996501354144`, `line`, `message_text`, `status="sent"`, `messenger_id`, `messenger_backend="wazzup"`

**Не входит:**
- Получение телефона и данных из Kommo API (будет в T03)
- Персонализация (реальные имя, дата термина) — будет в T03
- Логика окна времени 9-21 — будет в T04
- Cron-задача — будет в T04
- Telegram алерты — будет в T07

### Как протестировать

1. **Настроить .env:**
   ```bash

   WAZZUP_CHANNEL_ID=1b689b43-f846-41a6-bedc-cae01209fb8b
   WAZZUP_TEMPLATE_GUID=38194e93-e926-4826-babe-19032e0bd74c
   ```

2. **Запустить сервис:**
   ```bash

   ```

3. **Симулировать webhook от Kommo:**
   ```bash
   curl -X POST http://localhost:5000/webhook/kommo \
     -H "Content-Type: application/json" \
     -d '{
       "leads": {
         "status": [{
           "id": <тестовый lead_id из T00>,
           "status_id": <real_status_id "Принято от первой линии">,
           "pipeline_id": <real_pipeline_id "Бератер">

       }

   ```

4. **Проверить результат:**
   - WhatsApp +996501354144 получил сообщение с номера +49 3046690188

   - Кнопки: "Да, буду вовремя" / "Нет, не могу прийти"
   - SQLite: `SELECT * FROM messages` показывает запись со статусом "sent", `messenger_backend="wazzup"`
   - Response: `{"status": "ok", "message_id": "waba_msg_..."}`

5. **Негативные сценарии:**

## T04 — Ограничение времени отправки (9:00-21:00 CET/CEST)

**Дата:** 2026-02-23
**Статус:** draft
**Спецификация:** S01_whatsapp_auto_notifications.md

### Customer-facing инкремент

Сообщения, инициированные вне окна 9:00-21:00, откладываются до 9:00 следующего дня. Клиент не получает уведомления ночью.

### Scope

**Входит:**
- Функция `is_in_send_window()` → bool (проверка текущего времени в CET/CEST)
- Функция `get_next_send_window_start()` → datetime (следующее 9:00)
- Логика в webhook handler:
  - Если вне окна → `status="pending"`, `next_retry_at=9:00 завтра`
  - Если в окне → отправка сразу
- **Создание cron.py (базовая версия):**
  - Запуск каждый час (через systemd timer или cron)
  - Обработка `status="pending"` с `next_retry_at <= now`
  - Проверка окна времени перед отправкой
  - Отправка через messenger layer

**Не входит:**
- Логика повторов при отсутствии ответа (будет в T05)
- Проверка ответа от клиента

### Как протестировать

**Сценарий 1: Отправка в окне 9-21**
1. Установить системное время на 15:00
2. Переместить контакт на триггерный этап
3. Проверить: сообщение отправлено сразу

**Сценарий 2: Отложенная отправка**
1. Установить системное время на 22:00
2. Переместить контакт на триггерный этап
3. Проверить SQLite:
   ```sql
   SELECT status, next_retry_at FROM messages ORDER BY created_at DESC LIMIT 1;
   ```
   → `status="pending"`, `next_retry_at="2026-02-24 09:00"`
4. Изменить время на 9:00 следующего дня
5. Запустить `python cron.py`
6. Проверить: сообщение отправлено, `status="sent"`

**Сценарий 3: Переход на летнее время (CEST)**
1. Протестировать с датой в марте (CEST) и октябре (CET)
2. Проверить корректность окна 9-21

### Критерии приёмки

- [ ] `is_in_send_window()` корректно определяет окно 9:00-21:00 в CET/CEST
- [ ] Webhook вне окна создаёт запись со `status="pending"`
- [ ] `next_retry_at` устанавливается на 9:00 следующего дня
- [ ] Cron-задача обрабатывает pending сообщения и отправляет в окне 9-21
- [ ] Юнит-тесты для `is_in_send_window()` и `get_next_send_window_start()` проходят

### Зависимости

- **Требует:** T02 (webhook handler)
- **Блокирует:** T05 (повторы используют cron.py)
- **Можно параллельно с:** T03, T06, T07, T08

## T05 — Автоматические повторы (через 24ч, макс 2 раза)

**Дата:** 2026-02-23
**Статус:** draft
**Спецификация:** S01_whatsapp_auto_notifications.md

### Customer-facing инкремент

Если клиент не ответил на сообщение, система автоматически отправляет повтор через 24 часа. Максимум 2 повтора (3 отправки всего).

### Scope

**Входит:**
- Обновление логики webhook handler:
  - После первой отправки → `next_retry_at = now + 24h`
- Обновление cron.py из T04:
  - Получить сообщения: `status="sent"`, `next_retry_at <= now`, `attempts < 3`
  - Отправить повтор (тот же текст)
  - Обновить: `attempts += 1`, `sent_at = now`, `next_retry_at = now + 24h`
- Логика остановки повторов:
  - Если `attempts >= 3` → больше не повторять
- Повторы учитывают окно времени 9-21 (используют is_in_send_window из T04)
- Упрощение: повтор всегда (без проверки ответа от клиента)

**Не входит:**
- Проверка входящих сообщений от клиента (опция для будущего)
- Изменение текста сообщения при повторах

### Как протестировать

1. **Установить короткий интервал для теста:**
   - В `.env`: `RETRY_INTERVAL_HOURS=0.1` (6 минут)

2. **Отправить первое сообщение:**
   - Переместить контакт на триггерный этап
   - Проверить: сообщение получено

3. **Не отвечать на сообщение**

4. **Подождать 6 минут + запустить cron:**
   ```bash
   python cron.py
   ```

5. **Проверить:**
   - Второе сообщение получено (attempts=2)
   - SQLite: `attempts=2`, `next_retry_at` обновлён

6. **Повторить шаги 3-5:**
   - Третье сообщение получено (attempts=3)

7. **Ещё раз запустить cron:**
   - Четвёртое сообщение НЕ отправлено (достигнут лимит attempts=3)

### Критерии приёмки

- [ ] После первой отправки `next_retry_at` устанавливается на +24ч
- [ ] Cron-задача находит сообщения для повтора по условиям
- [ ] Повторное сообщение отправляется через 24ч
- [ ] `attempts` увеличивается на 1 после каждой отправки
- [ ] После 3 попыток сообщение больше не отправляется
- [ ] Повторы учитывают окно времени 9-21 (используют is_in_send_window из T04)

### Зависимости

- **Требует:** T02 (webhook handler), T04 (cron.py и окно времени)
- **Можно параллельно с:** T03, T06, T07, T08

**Спецификация:** S01_whatsapp_auto_notifications.md

  - POST `https://api.telegram.org/bot{token}/sendMessage`
  - Параметры: `chat_id`, `text`
- Интеграция в webhook handler и cron:
  - Try/catch при отправке через Wazzup24 → при ошибке вызов `send_telegram_alert()`
  - Try/catch при запросах к Kommo API
  - Try/catch в cron-задаче
- Формат сообщения:
  ```
  ⚠️ Ошибка отправки WhatsApp
  Lead ID: 12345

  Время: 2026-02-23 15:30
  ```
- Переменные окружения:
  - `TELEGRAM_BOT_TOKEN`
  - `TELEGRAM_ALERT_CHAT_ID`

**Не входит:**
- Telegram-бот для управления системой (команды /start, /status)
- Дашборд с метриками

2. **Симулировать ошибку Wazzup24:**
   - Указать невалидный `WAZZUP_API_KEY`
   - Переместить контакт на триггерный этап

3. **Проверить Telegram:**
   - Сообщение получено с деталями ошибки

4. **Симулировать ошибку Kommo API:**
   - Указать невалидный `KOMMO_TOKEN`
   - Переместить контакт

5. **Проверить Telegram:**
   - Сообщение об ошибке получения контакта

6. **Проверить логи:**
   - Ошибки логируются локально + отправлены в Telegram

### Критерии приёмки

- [ ] При ошибке Wazzup24 отправляется Telegram-алерт
- [ ] При ошибке Kommo API отправляется Telegram-алерт
- [ ] Сообщение содержит: lead_id, телефон, текст ошибки, время
- [ ] Ошибки в cron-задаче также генерируют алерты
- [ ] Алерты не блокируют основную логику (async или try/catch)

### Зависимости

- **Можно параллельно с:** T03, T04, T05, T06
- **Блокируется:** Требуется создать Telegram бота (см. S01, раздел "Зависимости")

## T08 — Деплой на Hetzner VPS (продакшн)

**Дата:** 2026-02-23
**Статус:** draft
**Спецификация:** S01_whatsapp_auto_notifications.md

### Customer-facing инкремент

Система работает на продакшн-сервере Hetzner, доступна по публичному webhook URL, обрабатывает реальные события от Kommo CRM.

### Scope

**Входит:**
- Dockerization:
  - `Dockerfile` (Python 3.11, копирование кода, ENTRYPOINT)
  - `docker-compose.yml` (сервис app + volume для SQLite)
  - `.dockerignore`
- Деплой на сервер 65.108.154.202:
  - SSH-подключение
  - Клонирование репозитория
  - Настройка `.env` (продакшн-значения)
  - `docker compose up -d`
- Настройка публичного URL:
  - Nginx reverse proxy: `https://sternmeister.example.com/webhook/kommo`
  - SSL-сертификат (Let's Encrypt)
- Настройка webhook в Kommo UI:
  - URL: `https://sternmeister.example.com/webhook/kommo`
  - События: смена этапа воронки
- Cron-задача на сервере:
  - Systemd timer или crontab: каждый час запуск `docker exec app python cron.py`

**Не входит:**
- Мониторинг (Prometheus, Grafana)
- Auto-scaling
- Backup SQLite (опция для будущего)

### Как протестировать

1. **Локальная сборка Docker:**
   ```bash
   docker build -t whatsapp-notifications .
   docker run -p 5000:5000 --env-file .env whatsapp-notifications
   curl http://localhost:5000/health  # → {"status": "ok"}
   ```

2. **Деплой на Hetzner:**
   ```bash
   ssh root@65.108.154.202
   cd /opt/sternmeister
   git pull
   docker compose up -d --build
   docker compose logs -f  # Проверить отсутствие ошибок
   ```

3. **Проверить webhook URL:**
   ```bash
   curl https://sternmeister.example.com/webhook/kommo
   ```
   → 405 Method Not Allowed (GET не поддерживается, нужен POST)

4. **Настроить webhook в Kommo:**
   - Открыть Kommo → Настройки → Webhooks
   - Добавить URL: `https://sternmeister.example.com/webhook/kommo`
   - События: "Смена этапа сделки"
   - Сохранить

5. **Тест end-to-end:**
   - Переместить реальный контакт на этап "Принято от первой линии"
   - Проверить: сообщение получено в WhatsApp
   - Проверить логи: `docker compose logs app`

6. **Проверить cron:**
   ```bash
   # На сервере
   docker exec whatsapp-notifications python cron.py
   # Проверить логи повторов
   ```

### Критерии приёмки

- [ ] Docker-образ собирается без ошибок
- [ ] Сервис запускается на Hetzner и доступен по HTTPS
- [ ] Webhook URL настроен в Kommo и принимает события
- [ ] Реальные события от Kommo обрабатываются корректно
- [ ] Cron-задача запускается каждый час (systemd timer настроен)
- [ ] SQLite база сохраняется между перезапусками (docker volume)
- [ ] Логи доступны через `docker compose logs`

### Зависимости

- **Требует:** T02-T07 (все функции реализованы и протестированы)
- **Блокируется:** Требуется настроить Nginx + SSL на Hetzner
